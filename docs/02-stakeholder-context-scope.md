## 1. Problem Frame (revised)

### 1.1 สถานการณ์ปัจจุบัน
ในปัจจุบัน เมื่อนักศึกษาหรือบุคลากรทำสิ่งของสูญหาย (เช่น บัตรประจำตัว กุญแจ สมุด หรืออุปกรณ์อิเล็กทรอนิกส์) หรือมีผู้เก็บสิ่งของเหล่านี้ได้ มักไม่มีจุดศูนย์กลางในการประกาศที่ชัดเจน ข้อมูลการแจ้งหายและพบของกระจายตัวอยู่ตามกลุ่มแชตหรือสื่อสังคมออนไลน์ต่างๆ ทำให้ข้อมูลถูกค้นหาได้ยาก และมีความเสี่ยงสูงที่จะมีบุคคลอื่นมาแอบอ้างเป็นเจ้าของโดยไม่มีหลักฐานยืนยัน

### 1.2 ใครได้รับผลกระทบ
* **ผู้ทำของหาย / ผู้สูญเสียทรัพย์สิน:** เดือดร้อนจากการสูญเสียทรัพย์สินหรือเอกสารสำคัญ และมีโอกาสได้ของคืนต่ำ
* **ผู้พบของ:** ต้องเสียเวลาในการตามหาเจ้าของเอง จัดการส่งคืนได้ลำบาก หรือไม่รู้ว่าจุดรับฝากอย่างเป็นทางการอยู่ตรงไหน
* **เจ้าหน้าที่ศูนย์รับของหาย:** ภาระงานเพิ่มขึ้นจากการต้องตอบคำถามซ้ำๆ และขาดระบบในการบันทึกและตรวจสอบของตกค้างที่ไม่มีข้อมูลบันทึก
* **ผู้บริหาร / ผู้กำหนดนโยบาย:** กระทบต่อภาพลักษณ์ของมหาวิทยาลัยด้านความปลอดภัย และอาจเกิดข้อพิพาททางกฎหมายหากการจัดการของตกค้างไม่ชัดเจน

### 1.3 Problem Statement
> ข้อมูลการแจ้งของหายและพบของในปัจจุบันกระจัดกระจายและขาดมาตรฐานในการพิสูจน์สิทธิ์ ทำให้การค้นหายากลำบาก เสี่ยงต่อการแอบอ้าง และอาจทำให้ข้อมูลส่วนบุคคลรั่วไหล

### 1.4 ผลลัพธ์ที่ต้องการ (โดยยังไม่กำหนด solution)
* มีกระบวนการแจ้ง ค้นหา ยืนยัน และส่งคืนสิ่งของที่ปลอดภัยและน่าเชื่อถือ
* มีการปกป้องข้อมูลส่วนบุคคลที่อาจติดมากับสิ่งของที่สูญหาย
* ลดขั้นตอนที่ยุ่งยากในการส่งมอบและรับคืนสำหรับทุกฝ่าย
* เจ้าหน้าที่มีเครื่องมือจัดการสิ่งของที่รับเข้า-จ่ายออกอย่างเป็นระบบ

### 1.5 สิ่งที่ทีมยังต้องเรียนรู้
* ข้อมูลใดของสิ่งของที่สามารถแสดงต่อสาธารณะได้ และเมื่อใดที่ควรปิดประกาศ
* ปัจจุบันมีวิธียืนยันตัวตนเจ้าของอย่างไร และจะยืนยันเจ้าของอย่างปลอดภัยได้อย่างไร
* ของประเภทใดที่ผู้พบต้องนำมาส่งให้เจ้าหน้าที่ทันที

### 2.1 Stakeholder Profiles

#### Stakeholder: ผู้ทำของหาย
* **Goal / need:** ค้นหาสิ่งของที่หายไปเจอได้อย่างรวดเร็วและตรงจุด ไม่ต้องไปไล่หาตามจุดรับฝากหลายๆ ตึก และยืนยันตัวตนเพื่อรับของคืนได้อย่างปลอดภัยโดยไม่ต้องเปิดเผยข้อมูลส่วนตัวในพื้นที่สาธารณะ
* **Pain point:** เดือดร้อนจากการสูญเสียทรัพย์สินหรือเอกสารสำคัญ (เช่น บัตร ปชช., กุญแจรถ)
* **Concern / risk:** กลัวว่าจะมีมิจฉาชีพมาแอบอ้างสวมรอยรับของที่มีมูลค่าไปแทน และกังวลว่าข้อมูลส่วนบุคคลจะหลุดรอดไปในโซเชียลมีเดีย
* **Open questions for Week 3:** ปัจจุบันมีวิธียืนยันตัวตนเจ้าของอย่างไร?

#### Stakeholder: ผู้พบของ
* **Goal / need:** ส่งต่อข้อมูลสถานที่พบและลักษณะสิ่งของได้อย่างสะดวก โดยไม่ต้องเก็บสิ่งของไว้กับตัว
* **Pain point:** เสียเวลาในการตามหาเจ้าของเอง หรือไม่รู้ว่าจุดรับฝากอย่างเป็นทางการอยู่ตรงไหน
* **Concern / risk:** ไม่อยากรับผิดชอบหากสิ่งของนั้นชำรุด สูญหาย หรือถูกกล่าวหาว่าขโมยมา และไม่ต้องการเปิดเผยช่องทางติดต่อส่วนตัว
* **Open questions for Week 3:** ของประเภทใดที่ผู้พบต้องนำมาส่งให้เจ้าหน้าที่ทันที?

#### Stakeholder: เจ้าหน้าที่ศูนย์รับของหาย
* **Goal / need:** มีเครื่องมือบันทึกข้อมูลสิ่งของที่รับเข้า-จ่ายออกอย่างเป็นระบบ ค้นหาง่าย และมีกระบวนการหรือหลักเกณฑ์ในการตรวจสอบหลักฐานผู้มารับของที่ชัดเจน
* **Pain point:** ภาระงานเพิ่มขึ้นจากการต้องตอบคำถามซ้ำๆ และต้องคอยดูแลทรัพย์สินของผู้อื่น
* **Concern / risk:** กลัวความผิดพลาดทางกฎหมายหากส่งมอบของให้ผิดคน และของตกค้างสะสมเป็นเวลานานจนล้นตู้โดยไม่รู้กำหนดเวลาทำลาย/บริจาค
* **Open questions for Week 3:** เมื่อใดที่ควรปิดประกาศหรือจัดการของตกค้าง?

#### Stakeholder: ผู้ดูแลระบบ IT
* **Goal / need:** รักษาระบบให้ทำงานได้อย่างเสถียร มีการจัดการสิทธิ์การเข้าถึงข้อมูลตามลำดับขั้น (Role-based access) และมั่นใจว่าระบบปฏิบัติตามนโยบายความเป็นส่วนตัว (Privacy/PDPA)
* **Pain point:** ต้องคอยตามแก้ปัญหาหากมีข้อมูลสำคัญรั่วไหลเข้าสู่ระบบ
* **Concern / risk:** ผู้ใช้อาจอัปโหลดรูปภาพสิ่งของที่เห็นข้อมูลส่วนตัวชัดเจน (เช่น เลขบัตร, ใบหน้าคนอื่น) เข้ามาในระบบสาธารณะโดยไม่เซ็นเซอร์

#### Stakeholder: ผู้บริหาร / ผู้กำหนดนโยบาย
* **Goal / need:** ต้องการลดข้อร้องเรียนเรื่องทรัพย์สินสูญหาย สร้างสภาพแวดล้อมที่ปลอดภัยในมหาวิทยาลัย และมีนโยบายจัดการสิ่งของที่ไม่มีผู้มารับ (Unclaimed items) ที่ชัดเจนและตรวจสอบได้
* **Pain point:** ภาพลักษณ์ของมหาวิทยาลัยด้านความปลอดภัยและการดูแลนักศึกษาได้รับผลกระทบ
* **Concern / risk:** ปัญหาข้อพิพาททางกฎหมายหากนโยบายการจัดการของตกค้างไม่ชัดเจนหรือละเมิดสิทธิ์ของนักศึกษา

### 3.1 System Boundary
ขอบเขตของระบบครอบคลุมการแจ้งของหาย การแจ้งพบของ การจัดการภาพถ่ายและคำบรรยายเพื่อความเป็นส่วนตัว การค้นหาและจัดหมวดหมู่ กระบวนการยืนยันความเป็นเจ้าของเบื้องต้น และระบบจัดการสถานะสิ่งของสำหรับเจ้าหน้าที่ ระบบ **ไม่ใช่** ระบบติดตามพัสดุ การเชื่อมต่อกล้องวงจรปิด (CCTV) การยืนยันตัวตนทางกฎหมายเต็มรูปแบบ หรือระบบการขายทอดตลาด/ประมูล

### 3.2 Key Data Flows
| From → To | Data / Request | Why it matters |
| :--- | :--- | :--- |
| **ผู้ทำของหาย** → ระบบ | ข้อมูลแจ้งของหาย, คำขอยืนยันสิทธิ์รับของ | จุดเริ่มต้นของการตามหาและจับคู่ข้อมูล |
| **ผู้พบของ** → ระบบ | ข้อมูลแจ้งพบของ, สถานที่พบ, ภาพถ่าย (ที่ถูกจัดการ privacy) | นำข้อมูลสิ่งของเข้าสู่ระบบเพื่อให้ค้นหาได้ |
| ระบบ → **ผู้ทำของหาย / ผู้พบของ** | รายการของหาย/พบของ (แบบเซ็นเซอร์ข้อมูลส่วนตัว) | ให้ผู้ใช้ค้นหาและตรวจสอบสถานะเบื้องต้นด้วยตนเอง |
| **เจ้าหน้าที่** ↔ ระบบ | อัปเดตสถานะรับเข้า/จ่ายออก, ตรวจสอบหลักฐาน | เป็นการจัดการของจริง (Physical) ให้ตรงกับข้อมูลในระบบ |
| ระบบ → **เจ้าหน้าที่** | แจ้งเตือนระยะเวลาจัดเก็บ (ของตกค้าง) | ช่วยบริหารจัดการพื้นที่และปฏิบัติตามนโยบายของสถาบัน |
| ระบบ ↔ **ระบบยืนยันตัวตนของมหาวิทยาลัย** | คำขอตรวจสอบบัญชีและข้อมูลยืนยันตัวตน | ลดความเสี่ยงจากการสวมรอยของมิจฉาชีพ |
| ระบบ → **ระบบแจ้งเตือน** | ข้อมูลเพื่อส่งข้อความ/อีเมลอัปเดตสถานะ | ทำให้ผู้ทำของหายหรือเจ้าหน้าที่ได้รับข้อมูลทันเวลา |

## 4. Scope Statement

### In Scope
1. ระบบการแจ้งของหายและการแจ้งพบของ
2. ระบบจัดการภาพถ่ายและคำบรรยายเพื่อความเป็นส่วนตัว
3. ระบบค้นหาและจัดหมวดหมู่
4. กระบวนการยืนยันความเป็นเจ้าของเบื้องต้น
5. ระบบจัดการสถานะสิ่งของสำหรับเจ้าหน้าที่ และระบบแจ้งเตือนระยะเวลาจัดเก็บ

### Out of Scope
1. ระบบติดตามพัสดุหรือการจัดส่ง
2. การเชื่อมต่อระบบกล้องวงจรปิด (CCTV)
3. การยืนยันตัวตนทางกฎหมายเต็มรูปแบบ หรือการดำเนินการสืบสวนทางกฎหมาย
4. การจัดการธุรกรรมทางการเงิน และการขายทอดตลาดหรือประมูล
5. การจัดการของหายภายนอกพื้นที่มหาวิทยาลัย

### Constraints
* ต้องใช้ชื่อบุคคล ข้อมูลจำลองเท่านั้น ห้ามเผยแพร่เลขบัตรหรือข้อมูลส่วนตัวจริงในการพัฒนา
* การพัฒนาเริ่มต้นด้วยการจำลองสิ่งของทั่วไปก่อน
* ระบบต้องมีแนวทางในการจัดการภาพหรือคำบรรยายที่ชัดเจน เพื่อป้องกัน Privacy Leak

### Assumptions

## 5. Privacy, Ethics, Security and Responsible AI

| ประเด็น | การตัดสินใจ/ข้อควรระวังใน Case นี้ |
| :--- | :--- |
| **Data minimization** | ไม่เผยเลขบัตร/ข้อมูลส่วนตัวจริง ใช้ข้อมูลจำลองเท่านั้น และหลีกเลี่ยงภาพที่เห็นข้อมูลส่วนบุคคลหรือใบหน้าคนโดยไม่จำเป็น |
| **Role-based access** | รักษาระบบให้ทำงานอย่างเสถียร มีการจัดการสิทธิ์การเข้าถึงข้อมูลตามลำดับขั้น เพื่อป้องกันไม่ให้ข้อมูลสำคัญรั่วไหล |
| **Transparency** | มีนโยบายจัดการสิ่งของที่ไม่มีผู้มารับ (Unclaimed items) ที่ชัดเจน ตรวจสอบได้ และสามารถติดตามสถานะสิ่งของได้ |
| **Responsible AI** | หากใช้ AI (เช่น Gemini) ช่วยในการคิด ให้บันทึกว่าใช้ช่วยด้านใด (เช่น การทำบทบาทสมมติ Stakeholder) และทีมต้องเป็นผู้ตรวจทานหาความถูกต้องของข้อมูลตาม Case Card เสมอ |
