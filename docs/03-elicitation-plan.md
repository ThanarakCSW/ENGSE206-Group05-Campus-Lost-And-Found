## 3. Technique Selection Rationale

### Why Interview?

การสัมภาษณ์เหมาะกับการเจาะลึกเหตุผล ประสบการณ์ และความกังวลที่แท้จริงของผู้ใช้ เช่น การทำความเข้าใจอุปสรรคที่ทำให้นักศึกษาไม่นำของไปส่งที่ศูนย์ (ตรวจสอบ AS-01) หรือการสอบถามวิธียืนยันตัวตนเพื่อคืนของมีค่าจากเจ้าหน้าที่การพูดคุยโดยตรงจะช่วยให้สามารถถามเจาะ (Follow-up) พฤติกรรมในอดีตได้

### Why Document Analysis?

คำถามที่เกี่ยวข้องกับข้อกฎหมาย นโยบายมหาวิทยาลัย (เช่น Data Retention, PDPA) หรือกฎการจัดการของตกค้าง (Unclaimed Items) จำเป็นต้องอิงจากเอกสารนโยบายที่เป็นลายลักษณ์อักษรจริงการใช้ Document Analysis ควบคู่กับการสัมภาษณ์ผู้บริหารและ IT จะช่วยลดความคลาดเคลื่อนจากความคิดเห็นส่วนตัว (Opinion)

### Why Workflow Walkthrough / Simulation?

งานรับฝากและส่งคืนสิ่งของเป็นกิจกรรมที่เกิดทางกายภาพ (Physical item management) การให้เจ้าหน้าที่ ลองทำให้ดู หรือเล่าทีละขั้นตอนผ่านสถานการณ์จำลอง (Scenario) เช่น มีคนมารับโทรศัพท์มือถือ จะช่วยเปิดเผยช่องโหว่ ข้อจำกัด และกระบวนการที่อาจหลงลืมไปหากใช้เพียงการสัมภาษณ์แบบถาม-ตอบปกติ 

### Why not use Questionnaire as the main technique now?

Questionnaire เหมาะกับการเก็บข้อมูลจากคนจำนวนมาก แต่ในระยะเริ่มต้นทีมยังต้องเข้าใจ workflow และกฎเชิงลึกก่อน จึงเลือก interview และ simulation เป็นแกนหลัก

## 4. Readiness for Week 04 Simulation

### Stakeholder roles needed

1. **ผู้ทำของหาย / ผู้พบของ (นักศึกษา):** ผู้ใช้งานหลักที่ต้องการค้นหาของและส่งมอบของ 
2. **เจ้าหน้าที่ศูนย์รับของหาย:** ผู้ปฏิบัติงานหน้างานที่คอยรับฝากและยืนยันการส่งคืน 
3. **ผู้ดูแลระบบ IT:** ผู้กำหนดสิทธิ์และดูแลข้อจำกัดด้านความปลอดภัย/ข้อมูลส่วนบุคคล 
4. **ผู้บริหาร / ผู้กำหนดนโยบาย:** ผู้ดูแลภาพลักษณ์และกำหนดกฎการจัดการของตกค้าง 

### Material / role card needed

- บทบาท เป้าหมาย ข้อมูลที่รู้ และข้อกังวลหลัก (Pain points / Risks) ของแต่ละ Stakeholder
- สถานการณ์จำลอง (Scenarios) เพื่อใช้ประกอบการสัมภาษณ์ เช่น:
  - **การเจอของมีค่า:** บัตรประชาชนหรือกระเป๋าสตางค์หล่น 
  - **การรับของคืน:** กรณีหลักฐานยืนยันตัวตนคลุมเครือ
  - **ปัญหารูปภาพ:** มีการอัปโหลดรูปภาพที่ติดข้อมูลส่วนบุคคลชัดเจน
- ตัวอย่างนโยบาย PDPA หรือ Data Retention มหาวิทยาลัย (จำลอง)

### Planned note-taking format

- จดหลักฐานในรูปแบบ: `Evidence ID | Source | Statement/Observation | Type | Related Open Question | Initial Interpretation`
- แยก **Fact / Opinion / Assumption / Solution Proposal** ชัดเจน
- ใช้คำพูดหรือข้อความสรุปที่ตรวจสอบย้อนกลับไปยัง stakeholder role ได้

### Team agreement for facilitation

- **ธนรัก (Interviewer):** จะเป็นผู้นำการสนทนา เน้นถามถึงอดีต (Past behavior) ไม่ถามชี้ช่อง (Leading question) หรือเสนอ Solution ล่วงหน้า เพื่อป้องกัน Confirmation Bias
- **นรบดี (Note-taker):** จะทำการจดบันทึกตาม Template ที่จัดเตรียมไว้ โดยไม่ด่วนตีความ Requirement ด้วยตนเอง
- ทีมจะทำการทวนสอบความเข้าใจ (Validation Step) โดยสรุปปัญหาและ Flow ให้ Stakeholder ฟังและยืนยันเสมอก่อนจบการจำลองสถานการณ์ เพื่อให้แน่ใจว่าไม่ได้คิดหรือตีความไปเอง
